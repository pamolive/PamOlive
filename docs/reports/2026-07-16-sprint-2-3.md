# Sprint 2 and 3 delivery report

Date: 2026-07-16

## Outcome

Mandatory first-login TOTP enrollment and the three-network Docker boundary are
implemented, tested, and ready for deployment. The work preserves the existing
credential and audit keyring introduced in Sprint 1.

## Sprint 2 — mandatory MFA

PAM-olive already stored TOTP configuration in `MFADevice`, encrypted through the
isolated keyring, and recovery-code hashes in `MFARecoveryCode`. The implementation
therefore avoids adding a duplicate `mfa_secret` or mutable enrollment boolean to
the user table. `User.mfa_enrolled` is an explicit derived property backed by a
confirmed TOTP device.

`PlatformSecurityPolicy.require_mfa_for_all_users` defaults to `True`. The enrollment
middleware redirects every unenrolled interactive account to `/mfa/setup/`; only
enrollment and logout routes remain reachable. This includes product administration,
technical administration, target-secret reveals, and SSH/RDP session creation.
Service accounts remain excluded from interactive login.

Enrollment generates a keyring-encrypted TOTP secret and QR code. A valid TOTP must
be confirmed before the device is activated. Ten random recovery codes are hashed
in the database, displayed once under `no-store` cache controls, and consumed once.
The administration policy page presents a prominent warning while the global
requirement is disabled.

## Sprint 3 — Docker segmentation

Compose now defines exactly three networks:

- `frontend` for published reverse proxies and the services they proxy directly;
- `internal` for Django, PostgreSQL, Redis, Celery, keyring, and broker control calls;
- `targets` for SSH/RDP broker egress.

PostgreSQL, Redis, Celery, and the keyring are not attached to `targets`. No
PostgreSQL, Redis, gateway, Guacamole, or guacd service port is published. Caddy
remains the only public entry point. `frontend` must not be marked as a Docker
internal network because Synology's Docker engine does not publish Caddy's bound
ports in that configuration.

`scripts/check-isolation.sh` discovers the active Compose project, starts a
disposable Alpine container attached only to `targets`, and fails if TCP/5432 on
PostgreSQL or TCP/6379 on Redis is reachable. GitHub Actions runs this check after
the full Compose stack becomes healthy.

Docker bridges are not a physical firewall. Production still requires a dedicated
gateway VLAN and destination/port allowlists for approved targets.
